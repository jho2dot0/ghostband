# AGENTS.md

Guidance for AI coding agents working in the Ghostband repo.

## What this project is

Ghostband v0 is a **single-file Python CLI** (`ghostband.py`) that turns a
structured, markdown-style **song spec** into an **instrumental MP3**. The
pipeline is:

1. Read a spec file (`specs/*.md`).
2. **Anthropic (Claude)** parses the spec prose into an ElevenLabs
   *composition plan* (JSON) plus metadata, via a large hand-tuned system
   prompt (`PARSER_SYSTEM_PROMPT`).
3. Deterministic post-processing strips lead-vocal tags and reconciles
   duration.
4. **ElevenLabs Music API** renders the plan into audio.
5. Outputs are written to a timestamped folder.

Ghostband only produces the **instrumental bed**. Lead vocals are added later
in Suno using artifacts generated by `spec_creation_prompt.md`. This
instrumental-only constraint is a core invariant of the whole project тАФ keep
it in mind for any change touching parsing, prompts, or tag handling.

## Layout

- `ghostband.py` тАФ the entire application. Organized into labeled sections:
  config constants, `PARSER_SYSTEM_PROMPT`, credentials, spec reading,
  Anthropic parse, post-parse normalization (vocal stripping + duration
  reconciliation), display, ElevenLabs generation, output writing, `main()`.
- `specs/example_spec.md` тАФ the canonical worked example of the spec format.
- `spec_creation_prompt.md` тАФ prompt template that has Claude author a spec,
  lyrics, and a Suno style block. Enforces the instrumental-only rules.
- `requirements.txt` тАФ `anthropic`, `elevenlabs`, `keyring`, `rich`.
- `README.md` тАФ user-facing docs and the full Suno workflow.

## Running it

```
pip install -r requirements.txt          # Python 3.11+
python ghostband.py specs/example_spec.md
```

Useful flags: `--dry-run` (parse + display, no ElevenLabs call тАФ **use this
for testing parser changes without spending credits**), `--no-confirm`,
`--output-dir DIR`, `--seed N`.

There is no test suite, linter config, or build step. Validate changes by
running with `--dry-run` against `specs/example_spec.md`.

## Target platform: Windows first

Development may happen on Linux/macOS, but **the primary runtime is Windows.**
Keep every change cross-platform, and prefer the Windows-native form when
giving commands:

```
py -m pip install -r requirements.txt    # or: python -m pip install ...
py ghostband.py specs\example_spec.md    # 'python' also works if on PATH
```

There is no shebang and `ghostband.py` is not marked executable, so always
invoke it via `python` / `py`, not `./ghostband.py`.

Windows-portability rules (most are already enforced in code тАФ keep them):

- **Use `pathlib.Path` everywhere**, never hard-code `/` or `\` or `os.sep`.
  All file I/O is already `Path.read_text` / `write_text` with
  `encoding="utf-8"`; keep the explicit encoding so behavior doesn't depend
  on the Windows default code page (cp1252).
- **Filenames must stay Windows-legal.** `_slug` strips everything outside
  `[a-z0-9_]` and the output timestamp uses `%Y-%m-%d_%H%M%S` (no colons) on
  purpose тАФ Windows forbids `: \ / ? * " < > |` and reserved names
  (`CON`, `NUL`, ...). Don't "improve" `_slug` to preserve raw titles, and
  don't add colons to the timestamp.
- **Line endings:** the repo has no `.gitattributes`, so a Windows checkout
  may rewrite `*.md` / `*.py` to CRLF. Spec files are read as UTF-8 text and
  the `\r` characters ride along into the prose sent to Claude тАФ a stray
  control char, which the ASCII-clean invariant warns against. Recommend
  adding a `.gitattributes` (`* text=auto`, `*.py text eol=lf`,
  `*.md text eol=lf`) and/or normalize line endings when reading the spec.
- **Keyring backend is Windows Credential Manager** on Windows (Secret
  Service on Linux, Keychain on macOS). To clear a bad key, look there under
  service `ghostband`.
- **Debug dumps land in the current working directory**, not the output
  folder: `ghostband_parser_debug.txt` (parse failures) and
  `ghostband_plan_debug.json` (ElevenLabs errors). Note `.gitignore` covers
  the latter but **not** `ghostband_parser_debug.txt` тАФ fix that before it
  gets committed from a failed run.

## Conventions

- **Single file by design.** Do not split `ghostband.py` into modules unless
  explicitly asked. Preserve the existing `# ----` section banners.
- **Python 3.11+**, `from __future__ import annotations`, modern type hints
  (`list[str]`, `int | None`), dataclasses.
- Standard library + the four declared dependencies only тАФ don't add new deps
  without a clear reason.
- User-facing output goes through **`rich`** (`console.print`, `Panel`,
  `Table`, `Text`) тАФ match the existing styling idioms.
- The script exits via `sys.exit(...)` with meaningful codes on fatal errors
  (e.g. `2` insufficient credits, `3` rate limit, `4` plan rejected). Preserve
  these and verify any new code against the actual exit calls in `main()`.
  Note: a **401 is not a fatal exit** тАФ it clears the stored key and
  re-prompts via the keyring flow, so don't convert auth failures to
  `sys.exit`.

## Critical invariants тАФ do not break

- **Instrumental-only.** No lyrics, lead vocals, ad-libs, spoken word, or
  singer names should survive into the composition plan. Wordless textures
  (oohs/aahs/hums, choir pads) ARE allowed as instrumental layers. This is
  enforced in two places that must stay in sync:
  - `PARSER_SYSTEM_PROMPT` (instructs Claude), and
  - the deterministic filter `is_vocal_track_tag` / `_VOCAL_TRIGGER_RE` /
    `_WORDLESS_SAFE_RE` / `strip_vocal_tracks_from_plan`.
  If you add a vocal-performance trigger word, also consider whether a
  wordless-safe override is needed, and update `spec_creation_prompt.md`.
- **Negative styles are never stripped** тАФ only positive globals and positive
  local styles are filtered.
- **Bar counts are the source of truth for duration**, not any `Total
  duration:` field. `reconcile_durations` computes the real total from bars +
  BPM and only *warns* on >100ms drift. Don't make `Total duration:`
  authoritative.
- **Parser must return JSON** with both `metadata` and `composition_plan`
  keys. `_strip_json_fences` tolerates markdown code fences.
- **Tags must stay ASCII-clean.** Non-ASCII characters can corrupt in transit
  to ElevenLabs and become control chars. The prompt and chord-progression
  rules call for ASCII hyphens, not em-dashes.

## Spec format (parser input)

Sections: `# Title`, `# Artist`, `# Global`, `# Sections`, `# Mix notes`.
Section headers look like `## Verse 1 - 16 bars` (hyphen or em dash, optional
explicit duration). See `specs/example_spec.md` and the format block in
`spec_creation_prompt.md` for the authoritative shape before editing parsing
logic.

## External services & secrets

- **Anthropic** model: `ANTHROPIC_MODEL` constant. **ElevenLabs** model:
  `ELEVENLABS_MODEL_ID`. Bump these at the constants, not inline.
- API keys are stored in the **OS keyring** (service `ghostband`, usernames
  `anthropic` / `elevenlabs`), prompted on first run, auto-re-prompted on 401.
  On Windows this is **Windows Credential Manager**. Keys are read with
  `getpass` (hidden input). Never log, print, or hardcode keys.
- Output dirs, `outputs/`, and `*_debug.*` files are gitignored тАФ don't commit
  generated audio, plans, or debug dumps.
